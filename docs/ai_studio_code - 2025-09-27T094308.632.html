UI/UX Design Brief: Portal ERP

1. General Information and Business Context

1.1. Project Goal

Creation of a comprehensive digital platform for automating the lease-to-own car leasing business. The platform is designed to dramatically reduce deal processing time, minimize operational costs, and provide clients and investors with a fully transparent digital service.

1.2. Key Business Problem to Solve

The company's current processes are constrained by manual operations, disparate tools, and lack of a unified data source. This leads to lengthy client onboarding, high operational costs, lack of control, and opacity for key stakeholders (clients, investors).

1.3. Core Design Principle

Proactivity, not Reactivity (Proactive, not Reactive). The interface should not just display information, but actively guide the user, highlight potential problems, automate routine actions, and suggest next steps. The design should be clean, intuitive, and focused on solving a specific task on each screen.

3. User Scenarios

Роль: Клиент (Алекс, 32 года, IT-специалист, экспат)

"Я только что переехал в Дубай и мне срочно нужна машина для работы и семьи. Банки отказывают в кредите из-за отсутствия кредитной истории, а процесс аренды непрозрачен. Мне нужна компания, которая предложит понятный путь к владению автомобилем. Я хочу зайти на сайт, быстро выбрать машину, загрузить документы со своего ноутбука, получить четкую обратную связь, если что-то не так, и отслеживать каждый шаг одобрения онлайн. Я не хочу звонить менеджерам или терять время в догадках."

Роль: Операционный менеджер (Мария, 28 лет)

"Мой день — это хаос из вкладок: CRM, почта, таблицы Excel, Odoo. Когда приходит новая заявка, я трачу первые 20 минут просто на то, чтобы собрать все данные в одном месте. Самая большая боль — когда клиент присылает нечитаемый скан, и я узнаю об этом только через день. Мне нужна система, которая покажет мне все новые задачи на одном экране, предоставит всю информацию по сделке в одном окне — включая историю авто из Odoo — и автоматизирует рутину, вроде отправки напоминаний. Я хочу решать проблемы, а не заниматься копипастом."

Роль: Инвестор (Дэвид, 55 лет)

"Я вложил средства в несколько автомобилей и хочу понимать, как они работают. Мне не нужны ежедневные операционные отчеты, но раз в неделю я хочу заходить в свой кабинет и видеть четкую картину: какие машины в лизинге, какова их доходность, есть ли просрочки. Мне нужен дашборд с ключевыми метриками (ROI, утилизация) и возможность при необходимости 'провалиться' до деталей по конкретному активу. Я ценю свое время и доверяю только данным."

5. Detailed Screen Descriptions (Screen Flow & Components)

5.1. Common Components

5.1.1. Screen: Login / Registration (ID: G-1)

  • Назначение: Обеспечить безопасный вход для существующих пользователей и простую регистрацию для новых.
  • Ключевые блоки: Контейнер с вкладками "Вход" и "Регистрация".
  • URL (Path): /login, /register
Компоненты (Вкладка "Вход"):
Элемент / Компонент Тип Описание и Поведение Источник данных / API Обязательность
Поле "Email" Input Text Валидация на корректный формат email. Автозаполнение. auth.email Да
Поле "Пароль" Input Password Маскирование ввода. Иконка для просмотра пароля. auth.password Да
Кнопка "Войти" Button (Primary) При клике: валидация, отправка запроса. Блокируется с показом спиннера во время запроса. POST /api/auth/login Да
Ссылка "Забыли пароль?" Link Ведет на страницу восстановления пароля (вне MVP). N/A Да
Компоненты (Вкладка "Регистрация"):
Элемент / Компонент Тип Описание и Поведение Источник данных / API Обязательность
Поле "Имя" Input Text Валидация на непустое значение. user.name Да
Поле "Email" Input Text Валидация на корректный формат и уникальность (асинхронная). user.email Да
Поле "Пароль" Input Password Валидация на минимальную сложность (8 символов, 1 цифра, 1 буква). user.password Да
Кнопка "Зарегистрироваться" Button (Primary) При клике: валидация, отправка запроса. Перенаправляет на онбординг. POST /api/users Да

5.1.2. Navigation (Sidebar/Header)

  • Назначение: Обеспечить глобальную навигацию по приложению и доступ к профилю пользователя.
  • Описание: В зависимости от роли, боковая панель (Sidebar) будет содержать разные пункты меню. Header содержит название текущего раздела, глобальный поиск (для Опер. менеджера) и меню пользователя.

Ключевые взаимодействия (Сценарий "Загрузка некачественного документа"):
  1. User uploads a blurry photo of Emirates ID.
  2. В блоке загрузчика появляется спиннер и текст "Анализ документа...".
  3. Через 2 секунды спиннер сменяется иконкой ошибки, появляется текст: "Ошибка: Фото слишком размыто. Пожалуйста, сделайте новый снимок." и кнопка "Загрузить другой файл".
  4. Прогресс по шагам не продвигается, кнопка "Отправить заявку" остается неактивной.
